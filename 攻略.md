# 書類審査の攻略

- `aidlc-docs/` の成果物を全て揃える（要件分析・ユーザーストーリー・アプリケーション設計・Unit of Work 計画）。「全て含む必要はない」とあるが「揃っていることが望ましい」
- ビジネス意図（Intent）を明確に言語化する。「なぜこのサービスが人をダメにするのか」のロジックを Requirements で説得力を持って書く
- Unit 分解の粒度を適切に。粗すぎず細かすぎず、実装可能性が伝わる分解にする
- Optional フェーズもやってスキルを見せる（ただし質が下がると減点リスク）
- audit.md を丁寧に残す。AI-DLC プロセスの実践度合いが審査観点になる

# アイデアの差別化

- 「ダメにする」の切り口を深掘りする。心理学的な依存メカニズム（可変報酬、損失回避、ソーシャルプルーフ等）を設計に組み込む
- ターゲットを絞る。万人向けより「エンジニアをダメにする」等ペルソナが明確だとストーリーが書きやすく審査でも伝わりやすい

# 技術面

- AWS サービスを幅広く使う構成にする（Lambda, DynamoDB, Bedrock, Step Functions, EventBridge, SNS/SQS, Cognito 等）
- Bedrock を活用して AI を組み込む。Bedrock Agents や Knowledge Bases を使うと AWS ネイティブ感が出る
- 予選（5/30）で動く MVP が必要。Inception 時点で実装可能性を意識した設計にしておく
- IaC（CDK/SAM）で再現可能な構成にしておくと決勝（AWS デプロイ必須）で楽

# プレゼン・決勝を見据えて

- QuizKnock が参戦する。プレゼンの面白さ・わかりやすさも重要
- 技術だけでなくストーリーテリングを意識する
