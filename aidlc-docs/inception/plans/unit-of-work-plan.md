# ユニット分割計画

## 概要
アプリケーション設計（C1〜C14）とユーザーストーリー（US-1.1〜US-9.2）を基に、開発ユニットに分割する。

---

## 分割方針の質問

### Q1: デプロイ単位
C5（画像生成 Python CLI）は独立ツールとして明確ですが、C3/C4/C6/C11/C14のLambdaツール群はどのようにデプロイしたいですか？

- A) 各ツールを個別Lambda（独立デプロイ）← **推奨**
- B) 関連ツールをまとめて1つのLambdaにグループ化
- C) 全ツールを1つのLambdaにまとめる

**推奨理由**: AgentCore GatewayのMCPツールは1ツール=1Lambdaが自然な構成。各ツールの責務が明確に分離されており、個別にテスト・デプロイ・スケーリングできる。コールドスタートも各Lambda軽量なので問題にならない。

[Answer]: A

### Q2: フロントエンド（C1）の分割
フロントエンド（Quart + Jinja2）は1つのユニットとして扱いますか？それとも画面単位で分けますか？

- A) 1ユニット（全画面まとめて）← **推奨**
- B) 画面グループ単位で分割（チャット系 / 図鑑系 / 認証系）

**推奨理由**: Quart + Jinja2のサーバーサイドレンダリングは1プロセスで動作し、ルーティング・テンプレート・静的ファイルを共有する。画面間の共通レイアウトやセッション管理もあるため、分割するメリットが薄い。

[Answer]: A

### Q3: エージェント（C2）の扱い
C2はStrands Agentsで、ツール呼び出しのオーケストレーションを担います。エージェント自体の開発ユニットはどう扱いますか？

- A) エージェント設定（プロンプト、ツール定義）を1ユニット ← **推奨**
- B) エージェント + フロントエンドをまとめて1ユニット（密結合のため）
- C) エージェントは独立ユニットだが、フロントエンドと同時開発

**推奨理由**: C2はAgentCore Runtime上で動作し、フロントエンドとは別のデプロイ単位。プロンプト設計・ツール定義・メモリ設定はフロントエンドのUI実装とは独立して開発・テストできる。

[Answer]: A

### Q4: インフラ（C7〜C10, C12, C13）の扱い
DynamoDB、S3、Cognito、Guardrails、Memory、LLMなどのインフラ/マネージドサービスの構築は？

- A) 1つの「インフラユニット」にまとめる
- B) 各サービスの利用コンポーネントに含める（C3がDynamoDBテーブルも管理する等）
- C) CDK/IaCで一括管理する独立ユニット ← **推奨**

**推奨理由**: DynamoDBテーブル・S3バケット・Cognito・Guardrailsは全コンポーネントが共有するリソース。CDKで一括定義すれば、テーブル設計変更時に1箇所で管理でき、環境ごとのデプロイも容易。各Lambdaユニットはリソース参照のみ行う。

[Answer]: C

### Q5: 開発優先順位
MVPの開発順序として、どのフローを最初に動かしたいですか？

- A) ダメ行動記録フロー（C1→C2→C11→C14→C1）を最優先で縦串貫通 ← **推奨**
- B) 認証（C9）を先に整備してからフロー開発
- C) 画像生成（C5）を先に完了させてからフロー開発

**推奨理由**: コアバリューである「ダメ行動→モンスター誕生」の体験を最速で検証できる。認証はCognitoで後付け容易、画像はプレースホルダーで代替可能だが、コアフローの体験は早期に確認すべき。

[Answer]: A

### Q6: バトルシステム（決勝用）の扱い
US-9.1/9.2のバトルシステムは決勝追加機能ですが、ユニット分割に含めますか？

- A) 含める（将来ユニットとして定義だけしておく）← **推奨**
- B) 含めない（MVP完了後に別途計画）

**推奨理由**: 定義だけなら低コストで、全体像の把握と依存関係の事前整理に役立つ。実装は後回しでも、ユニット境界を先に決めておけば他ユニットの設計時に考慮できる。

[Answer]: A
