# 要件明確化質問

## Question 1
ターゲットユーザーは誰ですか？

A) 20〜30代の社会人全般（性別問わず）
B) エンジニア・IT業界の人
C) 大学生・若年層
D) 特定のペルソナは設定せず幅広く
X) Other (please describe after [Answer]: tag below)

[Answer]: X

幅広い層をターゲットとするが、メインペルソナは20〜30代社会人。QuizKnockファン層（知的好奇心が高く自虐ネタを楽しめる層）も意識する。

**理由**: コンテスト決勝でQuizKnockが参戦するため、彼らのファン層にも刺さる設計にする。審査員に市場の広さを示しつつ、ユーザーストーリーはメインペルソナで具体的に書く。

## Question 2
MVP（予選デモ）のスコープとして、以下のどこまでを含めますか？

A) ダメ行動入力 → AI分析 → モンスター生成（テキスト情報のみ）
B) ダメ行動入力 → AI分析 → モンスター生成 + 画像生成
C) 上記B + 進化・変身システム + 図鑑
D) 上記C + ソーシャル機能（フレンド・通知）
X) Other (please describe after [Answer]: tag below)

[Answer]: C

**推薦理由**: 画像生成がないとデモのインパクトが弱い。進化・図鑑がないと「コレクションクエスト」の名前に合わない。一方ソーシャル機能は予選までの3週間では実装リスクが高い。Cが「見て楽しい＋遊べる」の最小構成。

## Question 3
ユーザーのダメ行動入力方式はどれですか？

A) フリーテキスト入力のみ
B) カテゴリ選択 + フリーテキスト補足
C) 音声入力対応あり
D) チャット形式（AIとの対話で記録）
X) Other (please describe after [Answer]: tag below)

[Answer]: X

チャット形式（AIとの対話で記録）。初回はフリーテキスト入力でユーザー独自の発想を引き出す。過去の入力履歴から選択肢ボタンを生成し、2回目以降はタップで素早く記録も可能にする。

**理由**: 初回テキスト入力でユーザーの個性あるダメ行動を収集し、モンスターのバリエーションを広げる。繰り返し入力は選択肢ボタンで簡易化し継続率を上げる。AgentCore Memory で過去入力を学習し、パーソナライズされた選択肢を提示。

## Question 4
モンスター画像の生成タイミングはいつですか？

A) ダメ行動を記録するたびにリアルタイム生成
B) 進化・変身時のみ生成（通常時はプレースホルダー）
C) 初回孵化時 + 進化時に生成
D) 事前に用意したテンプレート画像をバリエーション生成
X) Other (please describe after [Answer]: tag below)

[Answer]: C

**推薦理由**: 毎回生成（A）はコスト・レイテンシが高すぎる。進化時のみ（B）だと初回体験が弱い。Cなら「初めてモンスターが生まれる瞬間」と「進化の感動」の両方を画像で演出でき、API呼び出し回数も抑えられる。

## Question 5
認証・アカウント管理について、どの程度の機能が必要ですか？

A) メールアドレス + パスワードのみ
B) ソーシャルログイン（Google/Apple）対応
C) 匿名利用可能 + 後からアカウント紐付け
D) Cognito の標準的なサインアップ/サインインで十分
X) Other (please describe after [Answer]: tag below)

[Answer]: D

**推薦理由**: ハッカソンのMVPとしてはCognitoのデフォルト機能（メール+パスワード、確認コード）で十分。ソーシャルログインは設定工数が増える割にデモでの加点は少ない。Cognitoを使っていること自体がAWSサービス活用のアピールになる。

## Question 6
データの永続化について、ユーザーが記録したダメ行動の保持期間は？

A) 無期限（全履歴保持）
B) 直近1年分
C) ユーザーが削除するまで保持
D) 無期限 + エクスポート機能あり
X) Other (please describe after [Answer]: tag below)

[Answer]: A

**推薦理由**: モンスターの進化・図鑑は過去の全記録に基づくため、履歴を消すとゲーム性が壊れる。DynamoDBならストレージコストは低い。エクスポート機能（D）はMVPには不要な追加工数。

## Question 7
多言語対応は必要ですか？

A) 日本語のみ
B) 日本語 + 英語
C) 将来的に多言語対応を見据えた設計（MVP は日本語のみ）
X) Other (please describe after [Answer]: tag below)

[Answer]: A

**推薦理由**: AWS Summit Japan のハッカソンで、審査員も参加者も日本語話者。多言語対応の設計は工数に見合わない。「発表で英語が必要な場面はなし」と開催要項にも明記されている。

## Question 8
フロントエンドのプラットフォームは？

A) Web アプリのみ（モバイルブラウザ対応）
B) Web アプリ + PWA（インストール可能）
C) ネイティブモバイルアプリ（iOS/Android）
D) 企画書記載の通り Quart + Jinja2 による Web アプリ
X) Other (please describe after [Answer]: tag below)

[Answer]: D

**推薦理由**: 企画書で既に技術選定済み。Quart（非同期Flask）+ Jinja2 はPythonエコシステムで Strands Agents との統合が自然。SSRでSEO不要なハッカソンデモには十分。PWAは追加工数に見合わない。

## Question 9: セキュリティ拡張
このプロジェクトにセキュリティベースラインルールを適用しますか？

A) はい — すべてのセキュリティルールをブロッキング制約として適用（本番グレードのアプリケーション向け推奨）
B) いいえ — セキュリティルールをスキップ（PoC、プロトタイプ、実験的プロジェクト向け）
X) Other (please describe after [Answer]: tag below)

[Answer]: B

**推薦理由**: ハッカソンのプロトタイプであり、本番運用は想定しない。セキュリティルールをブロッキング制約にすると開発速度が落ちる。ただしCognito認証とGuardrailsによる入力フィルタリングは設計に含まれているため、最低限のセキュリティは確保される。

## Question 10: プロパティベーステスト拡張
このプロジェクトにプロパティベーステスト（PBT）ルールを適用しますか？

A) はい — すべてのPBTルールをブロッキング制約として適用（ビジネスロジック、データ変換、シリアライゼーション、ステートフルコンポーネントを持つプロジェクト向け推奨）
B) 部分的 — 純粋関数とシリアライゼーションのラウンドトリップのみにPBTルールを適用
C) いいえ — PBTルールをスキップ（シンプルなCRUDアプリ、UIのみのプロジェクト向け）
X) Other (please describe after [Answer]: tag below)

[Answer]: C

**推薦理由**: 進化判定ロジック等にPBTは有用だが、3週間のハッカソンでPBTをブロッキング制約にすると開発速度が大幅に落ちる。テストは通常のユニットテストで十分。
