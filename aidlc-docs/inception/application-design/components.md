# コンポーネント定義

## アーキテクチャ方針: エージェント中心型

エージェント（Strands Agents）がオーケストレーターとして全ビジネスロジックを担い、AgentCore Gateway 経由でツール（Lambda）を呼び出す。

---

## C1: Web フロントエンド（Quart + Jinja2）

**責務**:
- ユーザーインターフェースの提供（通常UI / ビジネスモード）
- AgentCore Runtime へのリクエスト送信とストリーミングレスポンス受信
- 演出（孵化・進化・技解放・UR）のレンダリング
- UIモード切り替え（通常↔ビジネス）

**インターフェース**:
- HTTP エンドポイント（ページ表示）
- AgentCore Runtime InvokeAgentRuntime API 呼び出し（HTTP streaming）
- S3 画像URL参照

---

## C2: ダメコレエージェント（Strands Agents on AgentCore Runtime）

**責務**:
- ユーザーとのチャット対話（深掘り質問、応答生成）
- ダメ行動テキストの分析・属性判定・深刻度算定
- モンスター生成/進化/技解放の判定とオーケストレーション
- AgentCore Memory によるユーザーパターン記憶・パーソナライズ
- Gateway 経由でツールを呼び出し

**インターフェース**:
- InvokeAgentRuntime API（ストリーミングレスポンス）
- AgentCore Memory（読み書き）
- AgentCore Gateway（ツール呼び出し）

---

## C3: 進化判定ツール（Lambda via Gateway）

**責務**:
- モンスターの進化条件チェック
- 分岐進化の判定（行動パターンに基づく進化先決定）
- レベルアップ計算

**インターフェース**:
- Gateway MCP ツールとして公開
- DynamoDB 読み書き（モンスターデータ、進化ツリー）

---

## C4: 技解放ツール（Lambda via Gateway）

**責務**:
- 必殺技の解放条件チェック
- 新技の生成（技名・効果をLLMで生成）
- 連続日数・組み合わせ条件の判定

**インターフェース**:
- Gateway MCP ツールとして公開
- DynamoDB 読み書き（技データ、記録履歴）
- C13 呼び出し（技名生成）

---

## C5: モンスター画像生成ツール（Python CLI）

**責務**:
- モンスター画像の事前生成（Gemini 3.1 Flash Image）
- 生成画像のS3アップロード
- 画像メタデータのDynamoDB登録

**インターフェース**:
- Python スクリプト直接実行
- DynamoDB 読み書き（画像メタデータ）
- S3 書き込み（画像アップロード）
- Gemini API 呼び出し（画像生成）

---

## C6: 図鑑ツール（Lambda via Gateway）

**責務**:
- 図鑑データの取得（一覧、フィルタリング）
- コンプリート率計算
- 隠しモンスターのヒント生成
- 進化ツリー情報の取得

**インターフェース**:
- Gateway MCP ツールとして公開
- DynamoDB 読み取り（図鑑データ、モンスターデータ）

---

## C7: データストア（DynamoDB）

**責務**:
- 構造化データの永続化（トランザクション系・マスタ系）
- オンデマンドキャパシティモード（スケール自動調整）

**テーブル設計**:

| テーブル | PK | SK | GSI | 主なアクセスパターン |
|---------|----|----|-----|-------------------|
| Users | `USER#<user_id>` | `PROFILE` | - | ユーザー情報取得 |
| Monsters | `USER#<user_id>` | `MONSTER#<monster_id>` | GSI1: `CATEGORY#<category>` | ユーザーの手持ち一覧、カテゴリ別検索 |
| DameActions | `USER#<user_id>` | `ACTION#<timestamp>` | - | ユーザーの記録履歴（時系列） |
| MonsterMaster | `MASTER#<master_id>` | `STAGE#<evolution_stage>` | GSI1: `ATTR#<attribute>` | 進化ツリー取得、属性別検索 |
| ImageMapping | `ATTR#<attribute>#STAGE#<stage>` | `RARITY#<rarity>` | - | 属性×進化段階×レア度→画像URL |

---

## C8: 画像ストレージ（S3）

**責務**:
- 事前生成済みモンスター画像の格納・配信
- 枠（フレーム）画像の格納・配信

---

## C9: 認証（Cognito）

**責務**:
- ユーザーサインアップ/サインイン
- JWT トークン発行・検証
- セッション管理

---

## C10: コンテンツフィルタ（Bedrock Guardrails）

**責務**:
- ユーザー入力テキストの安全性チェック
- 不適切コンテンツ（自傷・犯罪等）のブロック

---

## C11: 新規カテゴリ判定ツール（Lambda via Gateway）

**責務**:
- 新規カテゴリ判定（既存カテゴリか新規か）

**インターフェース**:
- Gateway MCP ツールとして公開
- DynamoDB 読み取り（モンスターデータ）

---

## C12: メモリ（AgentCore Memory）

**責務**:
- 対話コンテキスト・パーソナライズ情報の記憶

**保存対象**:
- ユーザーの行動パターン（よくやるダメ行動の傾向）
- 対話履歴・文脈（深掘り質問の流れ）
- パーソナライズ情報（口調の好み、反応パターン）
- 選択肢ボタン生成用の過去入力パターン

---

## C13: LLM（Bedrock Claude）

**責務**:
- テキスト生成（モンスター名、技名、応答文）
- ダメ行動の分析・属性判定
- 深掘り質問の生成

**インターフェース**:
- Bedrock InvokeModel API

---

## C14: モンスター生成ツール（Lambda via Gateway）

**責務**:
- 新モンスターの初期パラメータ生成（名前、属性、レア度）
- 画像選択（属性×進化段階×レア度→既存画像URL）

**インターフェース**:
- Gateway MCP ツールとして公開
- DynamoDB 読み書き（モンスターデータ、画像メタデータ）
- C13 呼び出し（モンスター名生成）

---

## 画像アセット仕様

### 生成方式
- Gemini 3.1 Flash Image でモンスター画像を事前生成
- プロンプトファイル（.md/.txt）を調整しながら Python CLI スクリプトで一括生成
- 生成した画像を S3 にアップロードし、DynamoDB に画像メタデータを登録

### 画像構成
- **キャラクター画像**: モンスター本体（透過PNG）
- **枠画像**: レア度別フレーム（N:白/R:青/SR:紫/UR:金）× 4枚（事前デザイン）

### 組み合わせ表（MVP想定）

| カテゴリ | 内訳 | 枚数 |
|---------|------|------|
| 通常進化（6属性 × 3分岐 × 3段階） | 各属性9枚 | 54枚 |
| 合体進化（属性の組み合わせ） | | 20枚程度 |
| 闇落ち進化（各属性1体） | | 6枚 |
| 特殊・シークレット | | 20枚程度 |
| **合計目標** | | **約100枚** |

※ 枠画像（N/R/SR/UR）は別途4枚。

### 生成ワークフロー
1. プロンプトテンプレート作成（属性・進化段階ごとの指示）
2. Python CLI スクリプトで Gemini API を呼び出し画像生成
3. 生成結果を確認、プロンプト調整→再生成（イテレーション）
4. 確定画像を S3 にアップロード
5. DynamoDB ImageMapping テーブルにメタデータ登録
