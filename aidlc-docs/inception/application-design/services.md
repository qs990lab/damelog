# サービス定義

## オーケストレーションパターン

エージェント中心型アーキテクチャのため、**C2（ダメコレエージェント）がオーケストレーター**として全フローを制御する。

---

## S1: ダメ行動記録フロー

**トリガー**: ユーザーがチャットでダメ行動を入力

```
C1(フロントエンド) → C2(エージェント)
    C2 → C10(Guardrails): 入力フィルタリング
    C2 → C2: ダメ行動分析（属性・深刻度判定）
    C2 → C5(モンスター生成ツール): 新規カテゴリ判定
    
    [新規カテゴリの場合]
        C2 → C5: モンスター生成
        C2 → C1: 孵化演出イベント + 新モンスターデータ
    
    [既存カテゴリの場合]
        C2 → C3(進化判定ツール): 進化条件チェック
        C2 → C4(技解放ツール): 技解放条件チェック
        
        [進化条件達成]
            C2 → C3: 進化適用
            C2 → C1: 進化演出イベント + 更新データ
        
        [技解放条件達成]
            C2 → C4: 技生成・適用
            C2 → C1: 技解放演出イベント + 新技データ
        
        [条件未達成]
            C2 → C1: 成長結果 + 進捗表示データ
```

---

## S2: 図鑑閲覧フロー

**トリガー**: ユーザーが図鑑画面を開く

```
C1(フロントエンド) → C2(エージェント)
    C2 → C6(図鑑ツール): コレクション取得
    C2 → C6: コンプリート率計算
    C2 → C1: 図鑑データ
```

---

## S3: 選択肢ボタン生成フロー

**トリガー**: リピートユーザーがチャット画面を開く

```
C1(フロントエンド) → C2(エージェント)
    C2 → AgentCore Memory: 過去の入力パターン取得
    C2 → C2: 選択肢ボタン生成
    C2 → C1: ボタンラベル一覧
```

---

## S4: 認証フロー

**トリガー**: ユーザーがサインアップ/サインイン

```
C1(フロントエンド) → C9(Cognito): 認証リクエスト
C9 → C1: JWT トークン
C1 → 以降の全リクエストにトークン付与
```

---

## 通信プロトコル

| 通信経路 | プロトコル |
|---------|-----------|
| C1 → C2 | HTTP streaming（InvokeAgentRuntime API） |
| C2 → C3/C4/C5/C6 | AgentCore Gateway（MCP / Streamable HTTP） |
| C2 → Memory | AgentCore Memory API |
| C2 → Bedrock LLM | Bedrock InvokeModel API |
| C2 → C10 | Bedrock Guardrails API |
| C1 → C9 | Cognito SDK |
| C3/C4/C5/C6 → C7 | DynamoDB SDK |
| C1 → C8 | S3 URL 直接参照 |
